# Waitlist + Clerk Gate Launch Runbook (Phase 4)

Manual setup steps to bring the new waitlist + sign-in + gated `/app/*` validation tooling preview live on `siliconcrane.com`. Code is in PR; this is the operator checklist.

**SC's dual identity:** the marketing landing keeps service positioning (sprint execution as the front door); the gated `/app/*` surface previews the validation tooling product. Both ship in this PR.

## 1. Cloudflare Pages SSR migration

The Astro site flips from `output: 'static'` to `output: 'server'` (still on `@astrojs/cloudflare` adapter). All existing pages (`/`, `/about`, `/services`, `/e/[slug]`, etc.) opted into `export const prerender = true` already, so they continue to be prerendered. Only `/sign-in`, `/app/*`, and any future Clerk-touching routes are dynamic.

- Verify the Cloudflare Pages project has SSR enabled (Workers binding present).
- Confirm `_routes.json` (auto-generated by the adapter) excludes prerendered paths from the worker invocation.

If `@clerk/astro` runtime issues appear on CF Pages (Phase 0 spike was inconclusive), fall back to migrating `apps/sc-web` to Vercel:

- `npm i @astrojs/vercel` and replace `cloudflare()` with `vercel()` in `astro.config.mjs`.
- Update CORS_ORIGIN in `workers/sc-api/wrangler.toml` if the Vercel preview URLs differ.
- sc-api Worker stays on Cloudflare regardless.

## 2. Clerk app provisioning

Clerk Pro (already paid for via DC) covers all four ventures.

- https://dashboard.clerk.com → New application → "Silicon Crane"
- Settings → Restrictions → enable **Allowlist mode**
- Allowlist → add `smdurgan@venturecrane.com` + any test invitees
- Settings → Email & SMS → Magic link → enable as primary sign-in method
- Customize the magic-link email template (sender: `hello@mail.siliconcrane.com`, body styled to SC voice — calibrated, evidence-driven)
- Save the Clerk publishable + secret keys in Infisical `/sc`

## 3. Resend domain verification

The existing sc-api code already references `noreply@siliconcrane.com` as a Resend sender for the leads/contact flow. The new waitlist code uses `hello@mail.siliconcrane.com` (subdomain pattern, isolated from the apex's existing Cloudflare Email Routing).

- https://resend.com/domains → Add domain → `mail.siliconcrane.com`
- Add the SPF + DKIM TXT records and DMARC record to Cloudflare (siliconcrane.com zone). Subdomain-scoped — apex Cloudflare Email Routing untouched.
- Wait for `Verified`.
- `RESEND_API_KEY` is already configured for sc-api (per existing code).

## 4. Cloudflare Turnstile

- https://dash.cloudflare.com → Turnstile → Add site

- `TURNSTILE_SECRET_KEY` for sc-api worker (set if not already): see step 5.

## 8. End-to-end verification

### Marketing + waitlist

1. Visit https://siliconcrane.com — landing renders, top nav present (Sign in link visible), service positioning intact.

### Service contact (existing flow, untouched)

7. Scroll to the bottom **Contact** form (`#contact`). Submit a test inquiry → existing `/contact` flow works as before.

### Clerk gate

8. Sign out everywhere first.
9. Visit `/sign-in` with an allowlisted email → magic link arrives → click-through completes sign-in → lands on `/app`.
10. `/app` shows the Validation Tooling preview (Experiments / Leads / Decisions cards + the "in active development" CTA).
11. Click the user button (top right) → sign out → confirm redirect.
12. Visit `/sign-in` with a non-allowlisted email → blocked at Clerk.
13. Visit `/app` directly while signed-out → middleware redirects to Clerk sign-in.

## 9. Reviewing waitlist signups

```bash
cd workers/sc-api
wrangler d1 execute sc-db --remote --command \
  "SELECT id, email, signed_up_at, status, utm_source, ip_country FROM waitlist_signups ORDER BY signed_up_at DESC LIMIT 50"
```

To promote: copy email → Clerk Allowlist → mark row as `invited`:

```bash
wrangler d1 execute sc-db --remote --command \
  "UPDATE waitlist_signups SET status = 'invited' WHERE email = 'foo@example.com'"
```
